# PROGRESS — R-S-Visualizations monorepo

This file tracks **monorepo-level** state. Each subproject keeps its own journal
(`gallery/PROGRESS.md`; `bundle/` once its spec arrives).

## What this branch did (`claude/monorepo-restructure`)

Converted the single root project (the R&S visualization gallery) into an
npm-workspaces monorepo of two sibling projects sharing a common math kernel.
**No second project was built** — `bundle` is an empty scaffold only, per the
instruction that its spec arrives separately.

### Repo name used

`R-S-Visualizations` — taken from the git remote
(`…/git/mesoscalar/R-S-Visualizations`). This is the value baked into the Vite
`base` paths:

- gallery → `/R-S-Visualizations/`
- bundle  → `/R-S-Visualizations/bundle/`

If the repo's Pages path differs, change `base` in `gallery/vite.config.ts` and
`bundle/vite.config.ts` accordingly (and the `bundle/` link in
`bundle/src/main.ts`).

### Decisions worth recording

- **Workspaces, not a path alias.** `@rsvis/math` is a real workspace package
  (its own `package.json` + tests). Its `exports`/`types` point at
  `./src/index.ts` (a barrel), so `tsc` (bundler resolution), Vite, and vitest all
  consume the TypeScript source directly through the hoisted `node_modules`
  symlink — no build step or emitted artifact for the kernel.
- **Single root lockfile.** The pre-existing root `package-lock.json` was
  regenerated by `npm install` to describe the workspace tree (it was *not* moved
  into `gallery/`). CI uses `npm ci` at the root.
- **`README.md` left at the repo root** (repo-level, not gallery-specific).
- **Combined Pages site, deploy from `main` only.** The workflow runs the `test`
  job on every branch (so PRs get a signal) and the `publish` job only on `main`
  (`if: github.ref == 'refs/heads/main'`), because the `github-pages` environment
  restricts deployments to the default branch. `publish` builds gallery + bundle,
  assembles `_site/` (gallery at root, bundle under `_site/bundle/`), and deploys.

### Nothing left to migrate

The widget kernels do not import one another, so extracting the five primitives
was clean and complete. The remaining `gallery/src/math/` modules are
gallery-specific applications and intentionally stay in the gallery (recorded in
`gallery/PROGRESS.md`). Promote any of them into `@rsvis/math` only if/when
`bundle` actually needs them.
